# HTTP 报文结构

http请求报文和响应报文有些不同

# 起始行

对于请求报文来说，起始行类似下面这样: 方法 + 路径 + http/版本

GET home HTTP/1.1

对于响应报文来说，起始行也叫响应行，类似: http/版本 + 状态码 + 原因

http/1.1 200 ok

# 空行

用空行来区分头部和实体

post 中，会先发送头部，响应 100 后，继续发送实体

# HTTP 的优缺点

HTTP 特点

1.灵活和可扩展

语义上的自由，只规定了基本格式（报文空行，换行分割字段），其他部分没有严格的语法限制。

传输形式的多样性(text,pdf,video)

2.可靠传输

基于tcp链接

3.请求-应答

一发一收、有来有回， 当然这个请求方和应答方不单单指客户端和服务器之间，如果某台服务器作为代理来连接后端的服务端，那么这台服务器也会扮演请求方的角色。

4.无状态

这里的状态是指通信过程的上下文信息，而每次 http 请求都是独立、无关的，默认不需要保留状态信息。

http 缺点

1.无状态

在需要长连接的场景中，需要保存大量的上下文信息，以免传输大量重复的信息，那么这时候无状态就是 http 的缺点了。

但与此同时，另外一些应用仅仅只是为了获取一些数据，无状态反而减少了网络开销，成为了 http 的优点。

2.明文传输

即协议里的报文(主要指的是头部)不使用二进制数据，而是文本形式。

这当然对于调试提供了便利，但同时也让 HTTP 的报文信息暴露给了外界，给攻击者也提供了便利。
WIFI陷阱就是利用 HTTP 明文传输的缺点，诱导你连上热点，然后疯狂抓你所有的流量，从而拿到你的敏感信息。

3.队头阻塞

当开启长链接时，多个请求共用一个`TCP`链接，当一个请求阻塞的时候，其他的请求只能等待

# Accept 字段

# 数据格式

Accept字段规定了接收方对实体部分数据的接受类型，

对于发送端而言类型体现在Content-Type

text： text/html, text/plain, text/css 

image: image/gif, image/jpeg, image/png 

audio/video: audio/mpeg, video/mp4 

application: application/json, application/javascript, application/pdf, application/octet-stream

# 压缩方式

采取什么样的压缩方式就体现在了发送方的Content-Encoding字段上， 同样的，接收什么样的压缩方式体现在了接受方的Accept-Encoding字段上

  • delate

  • br

  • gzip

# HTTP如何进行表单传输

在http中，有两种主要的表单提交的方式，体现在两种不同的Content-Type取值:

  • application/x-www-form-urlencoded
  • multipart/form-data

由于表单提交一般是POST请求，很少考虑GET，因此这里我们将默认提交的数据放在请求体中。

# application/x-www-form-urlencoded

对于application/x-www-form-urlencoded格式的表单内容，有以下特点:

  • 其中的数据会被编码成以&分隔的键值对

  • 字符以URL编码方式编码。

# multipart/form-data

对于multipart/form-data而言:

  • 请求头中的Content-Type字段会包含boundary，且boundary的值有浏览器默认指定。

例:

Content-Type:multipart/form-data;boundary=----WebkitFormBoundaryRRJKeWfHPGrS4LKe

数据会分为多个部分，每两个部分之间通过分隔符来分隔，每部分表述均有 HTTP 头部描述子包体，如Content-Type，在最后的分隔符会加上--表示结束。

# 小结

而且，在实际的场景中，对于图片等文件的上传，基本采用multipart/form-data而不用application/x-www-form-urlencoded，因为没有必要做 URL 编码，带来巨大耗时的同时也占用了更多的空间。
